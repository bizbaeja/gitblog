# **프로젝트를하면서 어려웠던 경험, 기억나는 경험을 왜 기억이나고 왜 힘들었고 어떻게 해결했는지 이야기해주세요**

- 프로젝트를 시작할 때 한 브랜치에서 시작했어야 하는데 그렇게 시작하지 못해서 중간에 컨플릭트 문제때문에 작업이 지연된적이 있었습니다. 제 코드가 따로 동떨어져있던 상황이어서 소통이 어렵게 된 상황이라 마음은 불안해져가고 있었던 상황이었고, 시간이 지연되면 다른 팀원들에게도 영향이 갈 수 있겠다는 생각이 들어서. 주변 시니어에게 도움을 청하게 되었고, 능숙하고 친절하게 설명해주신 덕분에 잘 해결된 경험이 있습니다. 때로는, 공식문서와 구글을 통해 빠르게 해결하는것이 베스트이지만 정말 급할때는 누군가에게 도움을 청할 줄 알아야 한다는 것을 깨달았습니다.

# 렌더링과 useState랑 연결해서 설명해보시오

- 리액트에서는 한번 렌더링되어 루트 내부에 삽입된 엘리먼트들은 화면을 새로고침 하거나 ReactDOM 을 다시 렌더링하지 않는 이상 업데이트되지 않습니다. 그래서 내부의 어떤 값을 변경하기 위해서 사용하는 함수가 바로 useState() 함수입니다.
- useState는 함수입니다. 이 함수를 이용해서 변수를 만들고 이 변수의 값을 변경해줍니다. 이렇게 변경된 값은 화면을 새로고침 하지 않아도 업데이트하여 반영해주는 것이 리액트의 특징입니다.
- 일반적인 변수 선언과는 다른 점이 있습니다. 첫 번째 배열 인자는 실제 저장할 데이터입니다.
- 두 번째 배열 인자는 변경할 함수의 이름을 만들어 주는 것입니다.

# (front) 프로젝트의 css를 작성하는데에 있어 bootstrap, tailwind 같은 프레임워크를 사용하셨다면 그이유를 설명해 주실수 있나요? 사용하지 않았다면 그 이유를 설명해 주실 수 있나요?

- 사용하지 않았다. 부트스트랩은 이미 완성되어있다는 느낌을 줄 수 있는 반면에 직접 css 를 한땀 한땀 작업하면서 깊은 이해도를 가지고 갈 수 있을 거라는 기대를 했기 떄문이다.

# (front) 자신이 만든 page중에 렌더링적으로 고민 해봐야 하는 부분이 있었나요 ? 있었다면 어떻게 해결을 하셨나요?

- useNavigate 와 a Link 를 적절히 사용하여 불필요한 렌더링을 최소화 시켰습니다. \_ useNavigate ⇒ 페이지 전환시 추가로 처리해야하는 로직이 있을 경우 바로 렌더링 없이 바로 화면전환 가능한 반면 Link 는 클릭시 바로 이동되며 페이지가 렌더링됩니다.

# 팀으로 협업을 이룬 프로젝트를 한 경험이 있으신가요? 있다면 협업을 이루는데에 자신에게 도움이 되었던것과, 제일 걸림돌이 되었던 것을 설명해 주세요

- 프로젝트 기획을 할 당시 처음에 할 말이 없을 때는 브레인 스토밍을 제안했었습니다. 주제를 하나 놓고 가지를 쳐가면서 이야기를 주고받다보면 어느새 좋은 아이디어가 생겨났던 경험이 있습니다. 논제를 펼쳐놓고 주장을 펼치는 것은 어렵지만. 단어 하나, 주제 하나를 던져놓고 상상을 펼치는 것은 덜 부담스러웠고, 그에 부응하여 빠르게 프로젝트 기획에 도달하게 되었던 경험이 있습니다.
- ‘현실적으로 어렵지 않을까’ 라는 질문 에 대한 대응
- 너무 공상으로만 가득한 의견이 아닌 이상은, 개발자로서 도전하는 것은 좋은 자세라고 생각합니다.웹소켓을 이용해 채팅기능을 구현하고자 하는 아이디어가 있었으나 그것의 구현에 있어서 너무 어려울것이라는 의견이 있었습니다. 부정적인 의견을 설득하고 북돋는 일은 생각보다 많은 에너지를 소모하는 일이었습니다. 다른 팀원이 우려하는 시간 부족에 대한 이슈를 해결하기 위해, 다른 기능을 줄이고, 소켓 기능을 집중했을 때, 우리의 서비스가 다른 팀들과 큰 차이점을 두게 될것이라고 설득했고, 결국 팀원들이 동의했던 경험이 있습니다.

# 로그인 회원가입을 어떻게구현되는지 아는대로 설명해보세요(토큰이 언제생성되고 언제 어떻게 주고받는지가 중요함)

- 사용자 로그인 요청 → 서버 사용자 확인 → DB (AT,RT 발급)
- 서버 (응답 AT,RT)→ 사용자에게 전달
- 사용자(AT 헤더에 실어 보냄) →서버에게 데이터 요청 (서버 AT 검증)
- AT가 만료된 상황 , 사용자 AT발급요청(AT 만료,RT) → 서버에 데이터 요청 (서버 AT 만료 확인 , RT 검증)
